## 1. 冻结结论

| 案例 | 冻结标识 | 规范／实现对象 | 项目配置 | 结论 |
|---|---|---|---|---|
| 《星际争霸》 | `SCR-BW-12310-13515-ELDRITCH-2P-TVZ-PROVISIONAL` | Blizzard 当前非中国区版本清单所列 `1.23.10.13515` 构建配置；官方地图归档中的 `(2)Eldritch Lake.scm`；原版官方手册与 Classic Battle.net 规则参考 | Windows；*Brood War*；两名人类；Terran 对 Zerg；`Melee`；客户端名义速度 `Fastest`；无 AI、观察者、联盟、共享视野、模组、EUD 或 UMS | **来源与地图已冻结；客户端可执行文件未取得、实机未复现，因此构建冻结为暂定** |
| 《外交》 | `DIP-HASBRO-4E2000-7P-STANDARD-GM` | Hasbro 官方静态英文 PDF，封面明确 `4th Edition 2000` | 标准七强国与初始配置；抽象座位 A–G 依规则列举顺序绑定七个强国；一名中立主持人；采用官方时限；不启用少人变体、缩短局、软件裁决或协商和局 | **规则对象与结构夹具已冻结；具体地图实物、实际玩家身份与对局序列未冻结** |

这组对照不是为了证明两款游戏“本质相同”，而是为了拆开几个经常被同一个词遮蔽的问题：

1. 实时世界中多个过程的重叠，不等于秘密指令的同步提交与批量裁决。
2. 玩家说出意图，不等于形成规则系统将要执行的正式**行动指令**。
3. 动态遮蔽当前世界状态，不等于暂时保密未来行动。
4. 软件持续执行规则，不等于人类按书面规则裁决整批冲突。
5. `Supply`、`Control`、`order` 等来源词必须先定位其角色、时点和关系，再映射到共享术语。

### 2.5 **命令生命周期**、时间与执行者

本案暂用以下结构描述，而不把它误写成“玩家执行了整段动作”：

```text
选择行动实体
  → 发出带目标／位置的玩家输入
  → 程序接收为行动指令（或追加到队列）
  → 程序在世界持续更新中尝试执行
  → 完成、因状态改变而失败，或被后续指令中断／替换
  → 产生位置、存量、实体或生命状态变化
```

- **[来源事实]** 出版者把产品归为 `real-time strategy`；手册允许在世界运行时发出命令，并让采集等过程自动重复。
- **[结构推导]** 世界中的移动、采集、建造、生产和战斗可以在玩家没有逐个触发每一步时继续变化，形成**重叠过程**。
- **[研究判断]** 本文称其为“程序更新的实时世界”或“过程重叠”，不称“所有事件真正并行”。
- **[文本／版本缺口]** 官方公开来源没有冻结 `1.23.10.13515` 的模拟帧时长、事件排序、网络锁步、随机数、寻路或同帧冲突决议。不得用玩家社区常用的帧数表补写。
- **[行为待证]** `micro`、`macro`、`APM`、地图控制、开局、骚扰或资源优化是可能的行为分析对象，不是仅凭规则文本已经观察到的机制事实。

### 3.3 阶段与批次结构

标准年度可写成：

```text
春季外交
  → 春季秘密写令
  → 同步公开并批量裁决
  → 必要时秘密撤退令、同步公开并裁决
  → 秋季外交
  → 秋季秘密写令
  → 同步公开并批量裁决
  → 必要时撤退
  → 秋季补给中心控制检查
  → 秘密增建／裁减令、同步公开并执行
  → 下一年
```

| 结构问题 | N1 直接规定 | 可作出的结构推导 | 仍不能推出 |
|---|---|---|---|
| 外交 | 玩家可公开或秘密交谈、组群、提出联盟、交换书面或口头承诺；协议不具规则约束力 | **非约束沟通**可以改变玩家信念，却不直接移动棋子 | 玩家一定会说谎、结盟或背叛；承诺怎样影响实际选择 |
| 写令 | 每名玩家秘密写下各单位的命令 | 命令单是未来行动的私有**承诺记录** | 玩家构思过程、草稿或口头计划已成为正式命令 |
| 公开 | 所有人完成后同时公开、宣读并核验命令 | 公开时点把私有未来指令转为公共输入；公开前不能按先后顺序观察别人再改令 | 现实操作能做到物理意义上的同一毫秒 |
| 合法性 | 合法但写错意图的命令仍执行；非法、含混或无法解释的命令不执行；无命令默认固守 | 文本形式、可解释性与规则合法性决定是否进入裁决 | 每一种边界歧义必有唯一无需裁判判断的答案 |
| 裁决 | 公开后由玩家或指定主持人解决全部冲突，结果为成功、失败、僵持、被击退等 | 单位并非按命令单宣读顺序逐个移动，而是以同批输入计算相容结果 | 人类实际裁决永不出错；存在官方机器算法或完整形式证明 |
| 撤退 | 被击退单位以秘密书面撤退令处理，再同步公开；撤退阶段不谈判；无合法撤退则解散 | 撤退是依赖前一批结果的新批次，不是原移动命令的自动尾部 | 玩家会选哪条撤退路径 |
| 调整 | 秋季控制检查后，根据补给中心与单位数增建或裁减；调整令秘密写下并同步公开，期间不谈判 | 空间控制经检查转换为下一年度的单位容量变化 | 长期扩张策略或某中心的主观价值 |
| 失序 | 缺少命令时单位全部固守且不提供支援；被击退者解散；不增建；必要裁减按规则回退 | 缺失输入不会暂停整个游戏，而会触发确定的**默认输入／回退规则** | 某玩家为何缺席，或这种回退是否公平 |

### 3.4 正式**命令生命周期**

```text
谈判、计划、承诺（规则上不约束）
  → 玩家秘密写入单位命令（正式输入）
  → 截止后同步公开（输入集合封闭并转为公共）
  → 核验合法性与可解释性
  → 对整批命令裁决冲突、支援、护航与僵持
  → 更新单位位置／被击退状态
  → 必要时开启撤退批次
  → 秋季再把控制结果转换为调整批次
```

N1 给出四类基本命令：`Hold`、`Move`、`Support`、`Convoy`。本资料包只冻结其结构作用，不重写整份裁决手册：

- `Hold`：单位不尝试进入另一省份；它仍可能成为被攻击或被支援对象。
- `Move`：单位尝试进入其类型允许到达的相邻省份；陆军与舰队受不同空间关系约束。
- `Support`：一个单位为另一单位的固守或移动增加冲突力量；支援必须满足可达关系，并可能被切断。
- `Convoy`：海上舰队链使陆军跨海移动；它把多个单位的相容命令与空间路径组合成一个条件性结果。

**[结构推导]** `Support` 和 `Convoy` 都不是玩家之间一般意义的合作关系，而是命令批次内具有明确参数、空间前提和失效条件的**关系效果**。玩家可以在没有可信联盟的情况下写出支援，也可以承诺支援却在正式命令中不这样做。

### 3.5 **信息访问**、秘密与公开

| 信息项 | 公开／秘密状态 | 何时改变 | 观察后效 | 证据边界 |
|---|---|---|---|---|
| 当前地图、单位与既成控制 | 原则上对所有玩家公开 | 批次裁决、撤退与秋季检查后更新 | 公共棋盘持续记录当前结果 | 不证明每名玩家记忆或理解都正确 |
| 谈判内容 | 可公开、私下或分组交流 | 由玩家选择传播对象 | 收件人可记忆或记录，但规则不保证真实性 | 不把私聊自动视为正式游戏状态 |
| 口头／书面协议 | 可由当事人知晓或公开，但不具规则约束力 | 后续谈判可修改、否认或违背 | 可能影响信念，不能强制未来命令 | 实际可信度、欺骗率与社会规范均为**行为待证** |

### 3.6 人类作为规则执行者

- **[来源事实]** N1 p. 4 允许一名中立主持人计时、收集与宣读命令、解决问题并作出裁决；p. 17 也允许玩家共同解决公开命令的冲突。
- **[结构推导]** 在本夹具中，主持人不是额外竞争玩家，而是**规则执行角色**：接收输入、封闭批次、解释文本、计算结果并写回公共状态。
- **[研究判断]** “规则系统”不必等同于软件。人类、纸面命令、时钟、棋盘和规则书共同构成此实例的执行架构。
- **[文本／版本缺口]** 规则要求主持人严格中立，却没有给出一套覆盖所有文本歧义、裁决错误、申诉或主持人更换的完整元程序。本夹具在发生真正无法由 N1 决定的争议时必须暂停记录，不得临时补房规后仍称标准规则。

| `convoy` | **协同运输指令链** | 是单位命令与海上路径的条件组合 |
| `supply center` | **单位容量来源节点** | 不是《星际争霸》的容量读数，也不是商品补给 |
| `unit` | **行动实体** | 单位类型决定可用空间关系 |
| `province`、`coast` | **规则空间节点**、**边界面** | 地理名称服务于离散移动规则 |
| `dislodged` | **被击退待撤状态** | 不是立即移动到任意相邻位置 |
| `standoff` | **等强冲突僵持结果** | 是批量裁决结果，不是谈判失败 |
| `control` | **秋季检查的中心控制状态**或**单位归属** | 必须标注对象与检查时点 |
| `build` | **调整期单位生成** | 不是连续空间施工过程 |
| `civil disorder` | **缺失输入回退规则** | 不应只按自然语言理解为社会混乱 |
| `Great Power`、`country` | **玩家控制阵营** | 不把历史国家整体等同于玩家本人 |

## 4. 跨案例校准

### 4.1 “同时”至少拆成三种结构

| 维度 | 《星际争霸》夹具 | 《外交》夹具 | 校准结论 |
|---|---|---|---|
| 输入窗口 | 世界更新期间持续接受玩家输入 | 固定谈判／写令窗口后封闭本批输入 | **持续输入**与**批次提交**不同 |
| 私密提交 | 对手不能直接读取玩家的未来队列，但来源未定义同步密封提交 | 每人秘密写令，截止后统一公开 | 只有后者明确具有**私有承诺—同步公开**协议 |
| 状态演化 | 多个自动过程在实时外观中重叠；底层帧排序未公开 | 本批命令先全部收集，再整体裁决 | **重叠过程**不等于**批量裁决** |
| 冲突处理 | 软件在世界更新中持续执行 | 玩家或主持人按书面规则解释整批冲突 | 执行者与可审计接口不同 |
| 后续批次 | 后续命令可随时中断／追加，具体实现待复现 | 撤退与调整是明示的新秘密批次 | 阶段边界的硬度不同 |

**[研究判断]** 暂不把“同时性”列成一个没有参数的原语。至少应描述：输入是否持续、是否有截止点、提交是否私密、何时公开、状态是持续更新还是批量求值、冲突由谁执行。

### 4.2 **命令生命周期**对照

| 生命周期槽位 | 《星际争霸》 | 《外交》 |
|---|---|---|
| 意图形成 | 玩家内部，规则不直接观察 | 玩家内部及谈判，谈判协议不约束 |
| 正式化 | 点击／热键向受控对象发令 | 写入秘密命令单 |
| 保存 | 程序的当前命令或队列 | 截止前的私有书面记录 |
| 可见性 | 自己获得界面反馈；对手只见可见世界结果 | 截止前秘密；随后全体同步公开 |
| 执行 | 程序持续尝试执行 | 主持人／玩家对整批命令裁决 |
| 失败 | 前提变化、中断、目标／路径等；精确语义待复现 | 非法／含混令不执行，冲突可失败或僵持 |
| 写回 | 世界状态持续改变 | 批量写回棋盘，再可能开启撤退批次 |

**[结构推导]** `command` 与 `order` 不能只按“都是动词指令”归为同一个无槽位原语。可复用的更小结构可能是：**意图**、**正式提交**、**承诺点**、**存储**、**公开**、**执行**、**失败**、**写回**。

### 4.3 两种**信息隐藏**

| 问题 | 《星际争霸》 | 《外交》 |
|---|---|---|
| 被隐藏的主要对象 | 当前世界中超出动态视野的敌方单位／更新状态 | 本批尚未公开的未来正式命令；部分私下沟通 |
| 访问条件 | 受控单位位置和视野关系 | 写令人身份、截止与公开阶段 |
| 持续时间 | 随空间关系持续变化 | 到同步公开点为止 |
| 历史残留 | 地形和建筑最后所见可保留，活动单位不持续显示 | 公开后的命令成为共同可核验输入 |
| 获取方式 | 探索、侦察、重新进入视野 | 到达制度化公开时点；私人话语由传播关系决定 |

**[研究判断]** **隐藏信息**必须至少带有“隐藏对象、主体、时段、访问关系、公开／更新条件、观察后效”这些槽位，否则会把侦察、记忆、秘密承诺与私人沟通压成一个词。

### 4.4 `supply` 与 `control` 的同词冲突

| 来源表达 | 实际规则角色 | 不能直接映射成 |
|---|---|---|
| 《星际争霸》Terran `Supplies` | 由特定实体提供、随单位占用的容量读数 | 可搬运的后勤物资；《外交》补给中心 |
| 《星际争霸》Zerg `Control` | 阵营特有的单位容量读数 | 领土控制；玩家控制组 |
| 《外交》`supply center` | 地图节点属性与秋季控制检查产生的单位容量来源 | 当前可消费库存；《星际争霸》UI 容量读数 |
| 《星际争霸》`control group` | 选择集召回绑定 | 所有权、容量或空间控制 |
| 《外交》`control` | 特定强国对补给中心的规则归属状态 | 玩家输入设备控制或动态视野 |

共同点只能在保留差异后表述：两案都存在约束单位规模的**容量结构**；前者由单位／建筑状态近实时更新，后者由领土节点在秋季检查后转成下一阶段调整权。词形相同不是共享语义，抽象相似也不能抹去拓扑和时序。

### 4.5 空间、资源、信息、时间与**机制编排**

| 维度 | 《星际争霸》 | 《外交》 |
|---|---|---|
| 空间 | 连续呈现的地图位置、可行走／可建造区域、距离与动态视野 | 离散省份、海域、海岸、邻接与单位类型可达性 |
| 资源 | 矿物／气体存量、单位容量、生产容量 | 补给中心产生的单位容量、调整资格、有限命令与时间窗口 |
| 信息 | 位置驱动的动态可见性、地形与最后所见残留 | 公开棋盘、私下谈判、秘密未来命令、制度化公开 |
| 时间 | 程序更新的实时世界、持续输入、自动过程重叠 | 年—季—阶段—批次的离散时序和截止点 |
| 执行 | 软件持续执行规则 | 人类玩家或中立主持人解释与裁决规则 |
| 编排骨架 | 采集 → 建造／生产／研究 → 单位与能力 → 空间行动／战斗 → 信息与资源反馈 | 谈判 → 秘密写令 → 同步公开 → 批量裁决 → 撤退 → 秋季控制与调整 |

**[研究判断]** 本对照暂不要求在“原语—机制—玩法模板—游戏”之间增加新层级。它要求把**编排**写得更精确：机制不仅是原语集合，还包含时序、信息通道、提交协议、执行者和写回关系。一个玩法模板可以编排多种机制；同一机制也可在不同时间制度与执行架构下产生不同玩法结构。

## 6. 冲突、缺口与后续复现需求

### 6.1 《星际争霸》

1. **客户端二进制**：取得官方 Windows 安装内容，记录安装器／主程序／关键数据文件的路径、字节数和 SHA-256，并从客户端内核对版本；在此之前保留 `PROVISIONAL`。
2. **启动配置**：确认当前 UI 中 *Brood War*、`Melee`、`Fastest`、观察者和出生点的实际可选方式，截图或录屏留证。
3. **地图结构**：解析或在编辑器中核对 `(2)Eldritch Lake.scm` 的起点、资源点与默认触发；不要把归档页的“推荐两人”替代地图内部结构。
4. **胜负路径**：实测建筑全灭、退出／投降、断线等不同终止路径；本轮只采用客户端宣告结果，不补写未核验快捷键。
5. **命令轨迹**：录制一段最小可控测试，分别覆盖立即命令、Shift 队列、采集循环、生产队列、前提失效和中断，并保存回放与视频。
6. **时间测量**：若后续需要帧／tick 主张，必须以该构建的可重复实验或官方技术材料为证，且将“观察到的客户端表现”与“内部实现解释”分开。
7. **行为材料**：若讨论 micro、macro、APM、开局或地图控制，另建行为证据包；不得从规则来源倒推。

### 6.2 《外交》

1. **地图制品**：扫描或取得与 4th Edition 2000 对应的官方地图，记录版本与字节身份；规则书描述足以分析规则，但不足以复现实体视觉布局。
2. **命令表单**：为研究局固定命令书写格式、截止记录、公开顺序和裁决记录表，但把项目记录协议与官方规则分栏。
3. **主持程序**：预先规定只按 N1 裁决；遇到 N1 无法决定的真歧义即暂停并记入失败日志，不临时创造房规。
4. **完整样例**：冻结至少一个包含支援切断、僵持、护航、被击退、撤退与秋季调整的官方或自制测试批次，逐步记录输入集合和裁决结果。
5. **行为材料**：若讨论承诺、欺骗、联盟、背叛或谈判风格，另行取得对局文本、录音授权或既有研究；规则只能证明这些交流被允许且不受约束。
6. **当前版关系**：若书稿需要谈 2021 F3155 或当前零售版，必须单独冻结英文规则制品，解决当前页面语言路由不稳定问题，不与 4th Edition 静默合并。

### 6.3 理论模型待检问题

- **命令**究竟是原语，还是由主体、对象、参数、提交、承诺、保存、公开、执行、失败和写回组成的句式？
- **同时性**是否应被建模为时间修饰，且至少区分重叠、同步截止、同步公开与批量求值？
- **信息隐藏**应放在状态、访问关系、时间修饰还是独立的观察语法中？
- **规则执行者**是机制内部槽位，还是游戏实例化时才选择的软件／人类实现？
- **容量资源**能否统一描述《星际争霸》的单位上限与《外交》的中心—单位关系，而不抹掉即时更新与秋季检查差异？
- **机制编排**是否已经足以说明玩法模板，还是仍需一种明确描述多机制耦合、反馈和主导循环的组合记法？

## 8. 移交摘要

本来源包允许后续标准案例安全地写出以下有限结论：

1. 两案都包含从玩家意图到规则状态变化的**命令生命周期**，但只有《外交》明确采用秘密承诺、同步公开和批量裁决。
2. 《星际争霸》的动态迷雾与《外交》的未来命令保密属于不同的**信息访问语法**。
3. 两案都限制单位规模，却分别通过实时更新的容量来源与秋季领土检查实现，不能因 `supply` 一词而合并。
4. 软件和人类都可以成为**规则执行者**；执行者、时间制度和写回方式应进入机制描述。
5. 本组尚未提供玩家行为或体验证据；也没有完成《星际争霸》客户端二进制和实机复现。后续案例必须保留这些边界。
