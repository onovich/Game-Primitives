# 游戏原语结构表示 v0.1

- 英文简称：Game Primitives Structural Representation 0.1
- 机器简称：`GP-SR 0.1`
- 状态：逻辑解谜先行组的冻结受测表示
- 日期：2026-07-23
- 来源：[理论地基](FOUNDATIONS.md)与 [ADR 0108](../docs/adr/0108-freeze-logic-pilot-execution-representation-and-rehearsal.md)

## 1. 用途与边界

GP-SR 0.1 是把来源规则转写成有类型、半形式结构描述的最小测试快照。它服务于 `logic-001` 的来源编码与独立重构，不是完整游戏描述语言、可执行 DSL、最终书稿定义或普适性证明。

本版本只回答：当前项目已经接受的规则句槽位与组合关系，能否在声明范围内承载并传递四类逻辑结构区别。它不描述玩家策略、难度、平衡、体验、美术、叙事或商业结构。

## 2. 可引用的基础类型

一个引用必须声明下列类型之一；类型是语义职责，不预先宣称每一类都是不可再分的原语：

| 类型 | 最小含义 |
| --- | --- |
| `entity` | 能被规则区分和引用的对象或代理 |
| `state` | 对象、关系、规则集或世界在某时刻可读取的值 |
| `relation` | 两个或多个引用之间有方向、角色或作用域的联系 |
| `condition` | 对事实、状态、关系或历史作出的布尔判断 |
| `action` | 代理可调用或系统可执行的操作 |
| `process` | 随时间或重复求值而展开的系统过程 |
| `rule` | 在声明作用域内把条件、行动与结果连接起来的规范结构 |
| `goal_evaluation` | 对完成、终止、得分、效用或其他评价的判断 |
| `feedback_observation` | 系统向某个观察者暴露的变化或信号 |
| `context_scope` | 限定规则适用的时间、空间、阶段、权限或信息范围 |

资源、位置、所有权、容量、时间和信息不是本版本中的无参数万能类型；它们必须由上述类型与显式角色表达。

## 3. 基本规则记录

每条规则只能使用以下槽位：

```text
在 Context 中，
当 Trigger 发生且 Preconditions 成立时，
Actor 可以或必须对 Target 执行 Action，
支付 Cost，
依 Resolution 得到 Effects，
系统产生 Feedback，
并据此更新 Goals / Evaluation。
```

| 槽位 | 允许内容 | 不得偷偷承载的内容 |
| --- | --- | --- |
| `context` | 时间、空间、阶段、权限、信息与规则作用域 | 未声明的全局默认值 |
| `trigger` | 使规则进入求值的事件、动作或状态变化 | 结算结果 |
| `preconditions` | 执行前必须成立的事实、许可与约束 | 动作后的可达性结论，除非它本身就是被检查的前置条件 |
| `actor` | 发起或负责动作的代理／系统角色 | 只因受影响就被称为行动者的对象 |
| `target` | 动作直接指向的对象、位置、关系或规则表达 | 未受当前动作指向的全部世界状态 |
| `action` | 被调用或执行的动作／过程 | 仅描述最终状态差分的名词 |
| `cost` | 动作承诺时消耗、锁定或冒险的内容 | 所有负面后果的笼统集合 |
| `resolution` | 冲突、随机、比较、排序或多步求值方法 | 没有说明顺序的结果列表 |
| `effects` | 动作结算造成的状态、关系、实体或规则集变化 | 玩家体验或策略推断 |
| `feedback` | 对指定观察者呈现的可观察信号 | 世界中存在但未被观察者取得的事实 |
| `goals_evaluation` | 完成、终止、得分、效用或其他评价更新 | 未经声明的玩家自定目标 |

没有语义内容的槽位可以为空，但必须显式标记为 `not_applicable` 或 `source_underdetermined`；不能依赖测试者从作品常识补全。

## 4. 组合关系

多条规则或记录只能使用以下八种已接受关系：

| 关系 | 含义 |
| --- | --- |
| `sequence` | 前一结构的结果成为后一结构的输入或触发 |
| `parallel` | 多个结构在同一作用域内同时求值或争用 |
| `nesting` | 一个结构只在另一结构的阶段或上下文内有效 |
| `gating` | 一个结构决定另一结构是否可用或生效 |
| `feedback` | 结果改变后续同类结构的参数或输入 |
| `mutual_exclusion` | 一项成立或被选择时关闭另一项 |
| `conversion` | 同一承载物在两个结构中改变语义角色 |
| `coupling` | 多个结构因共享状态而产生非局部影响 |

关系必须指向明确记录 ID，并说明方向、共享对象或生效条件。把若干机制名称并排列出不构成组合关系。

## 5. 来源编码状态

每个来源事实或必要规则边界必须记录一种状态：

- `encoded`：已由上述类型、槽位与组合关系表达；
- `source_underdetermined`：冻结来源无法确定；
- `representation_gap`：来源事实明确，但 GP-SR 0.1 无法承载；
- `scope_excluded`：事前排除在本轮范围外。

`representation_gap` 不能通过增加临时槽位、把答案写进备注或使用未声明的自然语言规则来消除。

## 6. 说明区

每个编码项目可以有一个说明区，但它只能记录：

- 来源定位与版本；
- 无法由来源确定的内容；
- 明确的范围排除；
- 身份泄漏风险；
- 编码与审核的未决异议。

凡会改变动作许可、状态转移、规则生效、目标判断或重构答案的语义，都必须进入正式槽位或登记为 `representation_gap`，不得只写在说明区。

## 7. 测试本地词汇边界

盲测材料可以使用**实体、状态、关系、动作、前置条件、效果、目标、作用域、时间、信息、规则生效**等基础结构词。不得在输入、ID 或说明中出现：

- **局部一致性**；
- **全局可完成性**；
- **规则变换边界**；
- 四种案例角色；
- `LC-01`–`LC-04`；
- 作品名、商标、原角色名或来源专有术语。

测试者可以使用不同表述。评价比较结构区别，不要求精确复述项目词汇。

## 8. 版本纪律

GP-SR 0.1 一旦随 `logic-001` 的 `frozen` 提交进入 Git，任何改变槽位、类型、组合关系、允许词汇或缺口处理的修订都必须发布为新版本，并进入新的正式测试轮次。格式修订若改变派发字节，同样不能覆盖已冻结输入。
