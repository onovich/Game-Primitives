# 游戏结构模型比较：从单向层级到多视图模型

- 状态：研究笔记 / 暂定结论
- 日期：2026-07-20
- 问题：`原语 → 机制 → 玩法模板 → 游戏` 是否足以作为《游戏原语》的核心模型？
- 证据范围：原始论文、作者或项目官方文档；访问日期均为 2026-07-20。

## 结论

**[暂定结论] 有必要改，但不必废弃。** 旧链准确抓住了“较小设计单位可以组合成较大结构”这一点，适合作为目录索引和构成视图的速记；它不足以同时解释游戏的构成、运行、体验、情境和设计过程。

建议把项目的核心模型改为三个相互关联的视图：

```text
一、构成视图（游戏制品内部）

有类型的原语 --语法--> 规则表达
规则表达 --组织--> 机制
机制 --耦合/编排--> 机制系统

二、运行视图（一次游玩中）

状态与反馈 → 玩家感知 → 决策/行动 → 规则求值与状态迁移
     ↑                                           ↓
     └────────────── 新状态与反馈 ──────────────┘

反复运行 → 动态与游玩轨迹 → 活动模式、策略与技能 → 体验与意义

三、情境视图（使游戏成为具体游戏）

机制系统 + 内容/关卡 + 表现/叙事 + 界面/物质实现
+ 参与者 + 平台与技术条件 + 社会文化情境
= 具体的游戏制品与游玩实践
```

“玩法模板”不再是机制上方的单一层级，而是**横跨上述视图的可复用设计假设**：它同时描述典型机制配置、玩家反复进行的活动与决策、时间和反馈结构、预期体验，以及成立所需的内容和情境条件。“游戏”也不是链条末端的一块更大积木；它是这些关系在特定实现与情境中的具体承载者。

一句话版本可以写成：

> 原语和语法构成规则，规则组织成机制与机制系统；机制系统在内容、界面、玩家与情境中运行，产生动态、玩法模式和体验；游戏是承载并实现这些关系的具体制品与实践。

## 旧链保留了什么，又压扁了什么

### 值得保留

- 它强调分析尺度：原语、机制、模式和具体作品不是同一个概念。
- 它强调组合而非关键词罗列，并为语法、类型检查和 DSL 留出了位置。
- 它把可复用设计结构与市场类型、题材和单个作品区分开来。
- 它容易教学：读者可以先从熟悉作品向下拆解，再尝试向上重组。

### 必须拆开的混合关系

1. **语法与机制之间缺少“规则表达”。** 语法规定表达是否合法；规则才规定条件、许可、动作、结算和状态变化。机制通常由一条或多条规则支持，不能把语法直接当成机制制造器。
2. **静态构成与运行时因果被混为一谈。** “机制由规则构成”是组成关系；“机制随玩家输入产生动态”是时间中的因果关系。两种箭头不能共用一个 `→`。
3. **“玩法模板”混合了设计配置和实际行为。** 机制组合是制品中的结构；策略、循环、诈唬、蹲点等则可能是玩家和系统共同产生的活动模式。前者可以写进规则，后者只能作为可预期、可观察、可检验的结果。
4. **“游戏”混合了制品与游玩。** 同一规则制品可以产生不同对局、社群规范和体验；同一玩法结构也能由棋盘、屏幕、身体或语言以不同方式实现。
5. **内容、界面、身体与情境无处安放。** 它们不只是最后的“包装”：界面决定玩家能感知和调用什么，内容与关卡改变机制实际形成的动态，物质和社会规范甚至参与构成规则。
6. **单向链容易被误读为设计流程。** 设计可以从期望体验、玩家活动、一个机制实验、题材材料或平台约束开始，并通过试玩返回修改任意层。

## 原始框架比较

| 框架 | 它主要表示什么 | 对本项目的关键提醒 | 没有充分覆盖什么 |
|---|---|---|---|
| MDA | Mechanics 是数据/算法与可操作组件；Dynamics 是它们响应玩家输入并相互作用时的运行行为；Aesthetics 是玩家的期望情绪反应。三者“彼此分开但有因果联系”；设计者与玩家从相反方向接近它们。 | 必须显式区分静态机制、运行动态和体验；跨层是因果与迭代，不是单纯的部件包含。 | Mechanics 边界过宽；内容、界面、文化情境不是完整层次。[Hunicke、LeBlanc、Zubek 2004，PDF pp. 1–5](https://cdn.aaai.org/Workshops/2004/WS-04-04/WS04-04-001.pdf) |
| Sicart 的机制定义 | 机制是“代理调用、用于与游戏状态交互的方法”；规则是规范性的，机制是施为性的。输入程序、状态、规则条件和体验之间存在映射。 | 可把机制暂定为“受规则约束、由代理调用的状态转换契约”，并明确 actor、precondition、cost、effect、feedback。 | 这是一种有意收窄的分析定义，不足以单独描述系统动态、内容和社会玩法。[Sicart 2008，§ Defining Game Mechanics / Conclusion](https://www.gamestudies.org/0802/articles/sicart) |
| Game Ontology Project | 以 interface、rules、goals、entities、entity manipulation 五个并列顶层组织设计元素；通过父子关系、组成关系、强弱案例和规则协同扩展本体。 | 原语应是多个有类型的词汇域和关系网，而不是一袋位于同一层的词；界面和目标不能被压进“机制”。研究应从典型、弱例和反例中 middle-out 地生长。 | 有意搁置大部分题材、表现和文化意义，也没有独立的策略、动态或体验层。[Zagal 等 2005，PDF pp. 2–11](https://doi.org/10.26503/dl.v2005i1.136) |
| Gameplay Design Patterns | 模式是“与 gameplay 有关的重复互动”；结构框架描述构件，模式描述玩家或计算机怎样使用构件影响游玩。模式之间有子模式、上位模式和冲突，不只是相加。 | “玩法模板”更像关系化、可复用的互动配置；应记录前提、后果、冲突、替代和实例，而非只列所含机制。 | 模式集合是工作语言，不是从原子严格演绎出的完备本体。[Björk、Lundgren、Holopainen 2003，PDF pp. 1、8–11](https://doi.org/10.26503/dl.v2003i1.60) |
| Ludii / ludemes | 游戏被表示为有类型、可嵌套并可执行的 ludemes；形式上区分 players、equipment、rules，rules 再分 start、play、end。一次 trial 是状态和动作序列。作者明确区分游戏的 form（规则与器材）和 play 中涌现的 function。 | 证明“有类型原语 + 语法 → 可执行规则描述”可行，也证明规则制品与运行轨迹必须分开；原语性取决于语言和抽象尺度。 | 主要覆盖传统策略游戏；能生成轨迹，却不直接表示身体体验、文化语境或玩家意义。[Piette 等 2020，PDF pp. 2–4](https://doi.org/10.3233/FAIA200120)；[Ludii Language Reference](https://ludii.games/downloads/LudiiLanguageReference.pdf) |
| Stanford GDL | 用 `role/init/true/does/legal/next/goal/terminal` 等关系定义有限状态、合法动作、迁移、终局和效用；GDL-II 再加入随机角色与玩家感知。 | 一套规则语言最低限度必须能回答：谁能行动、当前什么为真、什么动作合法、如何迁移、何时结束、如何评价、谁知道什么。 | 它描述可运行的游戏形式，不自动给出好策略、玩家活动模式、界面或体验。[Stanford GDL 规范，§ 2、5](https://logic.stanford.edu/ggp/notes/gdl.html)；[GDL-II，ch. 17](https://logic.stanford.edu/ggp/chapters/chapter_17.html) |
| Machinations | 以资源池、源、汇、转换器、门和连接模拟资源流；状态连接让当前资源反过来修改流量，便于观察反馈环、涌现和数值平衡。 | 机制“组合”应显式表示顺序、门控、资源流和正负反馈，并允许执行或模拟；结构图比成分清单更有解释力。 | 它有意聚焦内部经济，不能充当通用游戏本体或完整体验模型。[Dormans 2011，pp. 2–7](https://doi.org/10.1609/aiide.v7i3.12477)；[官方 Framework Basics](https://machinations.io/docs/framework-basics) |
| Operational Logics | 把抽象过程与玩家可理解的沟通/表征方式连接起来；仅有算法还不构成 operational logic，必须说明系统如何让特定受众把它理解为某个领域。 | 原语不仅要能执行，还要能被感知和解释；状态、操作、观察和呈现之间的接口是机制意义的一部分。 | 这是解释计算过程如何表意的分析单位，不直接提供完整游戏层级或通用设计流程。[Wardrip-Fruin、Mateas 2009，PDF pp. 3–5](https://doi.org/10.26503/dl.v2009i1.490) |
| Rules–Play–Culture | 游戏可从 rules、play、culture 三个理论模块观察；三者边界会模糊，并在“动态的效果架构”中相互关联。Play 是规则系统被玩家选择与行动启动后的经验。 | 完整研究不能把游戏还原成形式规则；玩家身体、社会实践、表现意义和文化位置需要独立视图。 | 它是多重分析透镜，不给出原语语法或可执行形式。[Salen、Zimmerman，Rules, Play and Culture，PDF pp. 2–5](https://ericzimmerman.com/assets/pdfs/RulesPlayCulture.pdf) |

这些框架并不互相取代。GDL/Ludii 回答“怎样使规则可执行”，Machinations 回答“系统流和反馈怎样展开”，MDA 回答“运行行为怎样联系体验”，GOP 与设计模式回答“怎样建立可比较的词汇”，Operational Logics 和 Rules–Play–Culture 则提醒我们：可执行结构仍不是玩家所理解和经历的全部游戏。

## 建议的工作定义

**[项目建议] 原语（Primitive）**

在声明的分析尺度和语言中，承担单一语义角色、具有类型与接口、暂不继续拆分的构件。候选域至少包括实体、状态/属性、关系、动作、条件、效果、事件、时间、信息、权限、目标、评价、反馈和界面映射。它是相对于模型的，不预设为宇宙中的绝对原子。

**[项目建议] 规则（Rule）**

由原语依照语法形成的规范或可执行表达，规定条件、许可/义务、动作、结算、状态迁移、信息变化、反馈与终止。规则是旧链最应补入的一层。

**[项目建议] 机制（Mechanic）**

代理或系统在规则约束下能够调用或反复执行的状态转换结构。机制可以由多条规则组成，也可与其他机制共享状态、互相门控和形成反馈。

**[项目建议] 机制系统（Mechanic System）**

多个机制通过顺序、并行、嵌套、门控、互斥、资源流、共享状态与反馈耦合形成的结构。这个概念承担旧链中“机制如何组合”的静态部分。

**[项目建议] 动态与玩法模式（Dynamics / Gameplay Pattern）**

机制系统在玩家行动和时间中展开后出现的状态轨迹、循环、策略、协作、节奏和涌现行为。应区分：

- **观察到的玩法模式**：从真实游玩或模拟轨迹中归纳；
- **预期的玩法模式**：设计方案对未来游玩的可检验假设。

**[项目建议] 玩法模板（Gameplay Archetype）**

跨层的复用模型，而非新的“零件层”。一张模板至少记录：

```text
结构签名：典型机制与耦合关系
活动签名：玩家反复做出的行动和决策
时间签名：循环、节奏、升级与终局
反馈签名：玩家看到什么、如何据此调整
体验承诺：希望支持何种感受、理解或社交关系
成立条件：内容、人数、界面、技能和情境假设
变体与反例：改变什么会使模板转化或失效
```

**[项目建议] 游戏制品与游玩实例（Game Artifact / Play Instance）**

游戏制品是规则和机制系统连同内容、表现、界面、物质与技术实现的具体安排；游玩实例是某些参与者在特定情境中实际运行它的过程。两者必须分别记录。

## 分析模型不等于设计流程

**[决定建议]** 项目应明确维护两套方向，避免把本体图误读成教程步骤。

分析既有游戏时：

```text
具体制品与游玩观察
→ 分别记录结构、运行和情境
→ 识别规则、机制、耦合、动态与活动模式
→ 用远缘案例和反例检验
→ 修订工作定义
```

设计新游戏时：

```text
设计意图或问题
→ 提出关于玩家活动/体验的假设
→ 选择规则、机制、内容与实现方案
→ 做出可玩原型
→ 观察真实动态、策略、理解和体验
→ 返回修改任意层
```

设计也可以从一个原语组合、题材材料或平台限制开始；关键不是起点固定，而是最终形成可测试的假设与反馈环。MDA 原文已经要求在系统、内容和体验各抽象层之间来回移动，并强调小改动会跨层级联；它不支持把 `M → D → A` 当成瀑布式生产流程。[Hunicke、LeBlanc、Zubek 2004，PDF pp. 1–2、4–5](https://cdn.aaai.org/Workshops/2004/WS-04-04/WS04-04-001.pdf)

## 当前建议与开放问题

**[决定建议]** 将旧链降级为教学速记：

```text
原语 --语法--> 规则 --组织--> 机制 --耦合--> 机制系统
```

把“玩法模板”“游戏”和“体验”移出这条纯构成链，交由运行视图、情境视图和跨层模板描述。不要再用同一种箭头同时表示组成、因果、涌现、实例化和创作顺序。

**[开放问题]**

1. “机制”是否只限玩家/代理可调用的方法，还是也包括自动运行的系统过程？
2. “玩法模式”与“玩法模板”是否需要成为两个正式术语，分别表示经验观察与设计假设？
3. 规则表达的最小必要槽位是什么；不同媒介是否需要不同句式？
4. 社会、私定、外部和物质规则应进入规则模型，还是只进入游玩情境？Stenros 与 Montola 将其与正式规则并列为五种规则，说明这个边界不能只靠电子游戏代码决定。[《The Rule Book》官方开放获取说明与内容摘要](https://mitpress.mit.edu/9780262377539/the-rule-book/)
5. 如何为“结构导致某种动态或体验”的论断记录条件、证据强度和失败案例？
