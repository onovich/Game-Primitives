# “移动”案例驱动研究试验

> 状态：开放研究笔记，不是项目定义或 ADR
> 日期：2026-07-20
> 研究问题：在不同游戏与分析尺度中，“移动”更适合被视为原语、机制、机制家族，还是随尺度变化的对象？
> 证据范围：规则书、原始论文、作者或项目官方文档；未进行玩家实验
> 项目约束：保持“原语 → 机制 → 玩法模板 → 游戏”的教学主线，不修改正式定义

## 结论摘要

**[综合判断]** “移动”不宜在不声明尺度的情况下被固定为一个本体层级。它至少有四种合法但不同的用法：

1. 在细粒度规则语言中，`位置`、`方向`、`相邻`、`速度`、`占据`、`重定位`等可以作为暂不继续分解的原语；
2. 在单个游戏的可执行规则中，“走一步”“滑行”“按骰点前进”“放置工人”“推箱”“驾驶”等是不同的机制或复合机制；
3. 在跨游戏目录尺度，“移动”最适合作为一个**暂定机制家族**：这些机制都改变实体相对于某个空间或拓扑的关系，但控制权、路径、时间、载荷与结算方式显著不同；
4. 当移动与目标、障碍、风险、节奏和反馈长期编排在一起时，平台跳跃、竞速、导航等才可能成为上层的玩法模板（Gameplay Pattern），而不是“移动”一词自身自动升级为玩法模板。

这个结论不否定项目当前的示例：[`Move(actor, destination)`](../../theory/FOUNDATIONS.md) 很适合表示“单实体、目的地已选、原子式重定位”这一子型。案例试验表明，它不应被误读为移动现象的完整定义。

真正施压定义的不是“还有很多种走法”，而是几类结构差异：国际象棋王车易位的一次行动改变两个棋子的空间状态（[FIDE 第 3.8.2 条](https://handbook.fide.com/chapter/e012023)）；《大富翁》经过起点与直接入狱产生不同路径语义（[Hasbro 规则第 2 页](https://www.hasbro.com/common/instruct/Monopoly_Vintage.pdf)）；《仓库番》一次推进行动耦合玩家与箱子并可能永久改变后续可达性（[Culberson 1997，第 1–3 页](https://sokoban.dk/wp-content/uploads/2016/02/Sokoban-is-PSPACE-complete.pdf)）；ORBITAL 中玩家调用的是吸引/排斥，飞行轨迹却由系统动力学生成（[Sicart 2008，ORBITAL 例段](https://www.gamestudies.org/0802/articles/sicart)）；Ludii 甚至把 `move` 用作一回合中的合法行动对象，其中未必包含空间位移（[官方语言参考第 8 章](https://ludii.games/downloads/LudiiLanguageReference.pdf)）。这些案例迫使研究者先说明“移动的是词、行动、状态差分，还是玩家所见的运动”。

## 1. 方法、案例角色与证据标签

### 1.1 与项目框架的关系

**[项目定义]** 本笔记沿用 [`CONTEXT.md`](../../CONTEXT.md) 与 [`FOUNDATIONS.md`](../../theory/FOUNDATIONS.md) 中的工作模型：原语是在声明尺度上暂不继续分解、承担一种基本语义角色的单位；机制由原语经语法组合，能够被触发、重复并产生可观察状态变化；玩法模板组织多个机制形成持续的行动、决策与反馈结构；游戏是具体实例。按照 [`ADR 0007`](../../docs/adr/0007-case-driven-research-spiral.md) 与 [`ADR 0008`](../../docs/adr/0008-use-multi-role-case-sets.md)，案例用于反复施压工作理论，而不是为既定答案挑选例证。

项目文档定义了**案例角色**，没有预先指定“移动”必须使用哪些游戏。因此，本试验严格按角色功能组建如下案例组，而不把游戏名单冒充项目决定：

| 案例角色 | 本试验中的实例 | 角色用途 |
| --- | --- | --- |
| 锚点案例 | 国际象棋的普通走子 | 熟悉、规则边界清楚，便于从 `Move(actor, destination)` 起步 |
| 对照案例 | 《超级马力欧兄弟》、F-ZERO、《大富翁》、Agricola | 分别改变时间模型、动力学、目的地产生方式、是否存在路径 |
| 纵深案例 | Zork/Inform 式文字冒险、《仓库番》 | 深挖输入解析与图拓扑、耦合推动与不可逆可达性 |
| 边界案例 | 王车易位/吃过路兵、《大富翁》直接入狱、ORBITAL、Ludii 的非空间 `move` | 寻找“移动”定义会失真或词义越界的地方 |
| 设计变体 | 第 8 节的六组规则突变 | 用最小可玩改动检验因果预测，而非只做术语辩论 |

覆盖范围包括棋类、平台动作、竞速、掷骰轨道桌游、工人放置、文字冒险/节点移动、推箱子/空间解谜。

### 1.2 标签

- **[来源事实]**：来源直接规定、描述或论证的内容；紧邻可点击的一手来源与页码/节号。
- **[项目定义]**：项目当前采用的工作定义或已记录决定。
- **[综合判断]**：本研究跨来源比较所得，不冒充来源作者原话。
- **[工作假设]**：可由原型或玩家研究证伪的预测。
- **[边界/反例]**：对某一候选定义构成压力，但未必足以推翻整个项目模型。
- **[开放问题]**：证据或尺度尚不足，暂不封口。

“玩家体验”一栏全部是**工作假设**。本试验没有访谈、遥测或对照实验，因而不把“紧张”“自由”“掌控感”等体验判断写成经验事实。

## 2. 先拆开“移动”：比较时使用的分析接口

**[综合判断]** 为避免把手柄按键、角色意图、合法性规则、坐标变化和屏幕运动统称为“移动”，每个案例都按七层记录：

| 层 | 本笔记中的问题 | 容易发生的混淆 |
| --- | --- | --- |
| 输入（input） | 玩家或系统提供了什么信号？ | 按左键不等于角色一定向左位移 |
| 行动（action） | 游戏把信号解释成什么可尝试的行动？ | 文本 `S` 与解析后的 `going south` 不是同一对象 |
| 规则（rule） | 什么条件下允许行动，如何结算？ | “车走直线”没有说明阻挡、吃子与将军约束 |
| 状态变化（state change） | 哪些变量实际改变？ | 位置之外还可能改变速度、回合权、历史权限、占据与分数 |
| 空间关系（spatial relation） | 空间是格、图、连续场、轨道，还是行动槽？ | 相邻不一定是几何相邻，也不一定存在可观察路径 |
| 系统动态（dynamics） | 多次规则执行与实时反馈如何随时间展开？ | 轨迹可能由物理/力场生成，而非玩家直接选择 |
| 玩家体验（experience） | 玩家可能感知到什么，如何验证？ | 动画“像移动”不证明规则层存在同名机制 |

这个接口只是研究工具，不是新增正式层级；它与 [`structural-models.md`](../foundations/structural-models.md) 中“不要把静态构成、运行因果与体验混成一种箭头”的问题意识相容，但不在此采纳该笔记提出的模型改写。**[来源事实]** MDA 原论文把 mechanics 视为数据与算法，把 dynamics 视为系统在运行时对输入的行为，把 aesthetics 视为期望的情绪反应；论文也以《大富翁》的骰子概率与 `Roll → Move` 反馈为示例（Hunicke、LeBlanc、Zubek，[原始论文 PDF，第 1–3 页](https://cdn.aaai.org/Workshops/2004/WS-04-04/WS04-04-001.pdf)）。

为横向比较，本笔记还记录一个**临时移动契约**：

```text
发起者 / 受控实体 / 输入与触发 / 起点 / 目的地或控制信号
空间拓扑与参照系 / 路径语义 / 时间模型 / 前置条件与成本
结算方式 / 耦合状态变化 / 反馈
```

**[综合判断]** 这不是对“移动”的定义，而是一张缺项检查表。某个案例可以没有玩家选择的目的地、没有可观察路径，甚至没有玩家直接调用的位移动作；缺项本身正是证据。

## 3. 锚点案例：国际象棋的普通走子

### 3.1 来源事实

**[来源事实]** FIDE 《国际象棋规则》规定棋盘为 8×8，双方交替行棋；每种棋子的合法去向不同，己方棋子占据的格不能进入，吃子作为同一步的一部分发生。象、车、后的线路不能越过阻挡棋子；马的去向不与原格同行、同列或同斜线；任何使己方王仍受攻击的行动都不合法（[FIDE Laws of Chess，2023 生效版，第 1–3 条](https://handbook.fide.com/chapter/e012023)）。

| 层 | 锚点分析 |
| --- | --- |
| 输入 | 实体棋局中拿起/触碰棋子并落到格上；数字实现可用点选或拖拽。输入媒介不是规则中的走法本身。 |
| 行动 | 选择一个己方棋子并尝试一个合法走子。 |
| 规则 | 棋子类型产生候选目的格；阻挡、占据、吃子与“不能令己王受将”再过滤合法性。 |
| 状态变化 | 通常改变一个棋子的位置；吃子同时移除对方棋子；回合权改变。 |
| 空间关系 | 离散方格；不同棋子使用不同关系：直线、斜线、马步、兵的方向性。 |
| 系统动态 | 多步交替使控制区域、威胁、阻挡与未来合法行动集合持续变化。 |
| 玩家体验 | **[工作假设]** 玩家会把走子体验为“选择去处及其战术后果”，而非操纵一段连续轨迹；可通过决策时间、候选格回看和访谈编码检验。 |

### 3.2 锚点能支持什么，不能支持什么

**[综合判断]** 对普通王一步或无阻挡的短距离走子，项目示例可以自然展开：

```text
Position(piece) + ReachRelation(pieceType) + Occupancy + KingSafety
  --合法性与结算语法-->
ChessMove(piece, destination)
```

但即使在这个最友好的案例中，`Adjacent(Position(actor), destination)` 也不是通式：车、象、后可跨越多格，马的可达关系不是通常邻接；“起点—终点”也不足以判断王是否被将。更稳妥的锚点结论是：**目的地选择与离散位置更新是一个清楚的移动子型；合法性来自多个原语与规则的组合。**

## 4. 对照案例：逐项改变锚点前提

### 4.1 平台动作：《超级马力欧兄弟》

**[来源事实]** 任天堂原版说明书把十字键列为移动，B 键列为跑步/火球，A 键列为跳跃/游泳；进入水管等情境还要使用特定方向输入。说明书同时规定落下、受伤、超时与关卡终点等后果（[Nintendo，《Super Mario Bros.》官方说明书，PDF 第 2–3 页](https://www.nintendo.co.jp/clv/manuals/en/pdf/CLV-P-NAAAE_en.pdf)）。

| 层 | 案例分析 |
| --- | --- |
| 输入 | 持续或瞬时的方向、跑、跳等按键组合。 |
| 行动 | 游戏解释为走/跑、起跳、游泳、进入水管等情境行动；“按右”不是选定一个目的格。 |
| 规则 | 控制状态、碰撞、地面/空中情境、危险与关卡边界共同限制行动结果。 |
| 状态变化 | 位置随时间多次更新，并可伴随运动状态、生命、时间、关卡进度改变。 |
| 空间关系 | 屏幕呈现连续位移；可站立表面、障碍、坑与水管赋予空间功能。 |
| 系统动态 | 输入保持时间与实时碰撞反馈形成轨迹；同一初始方向输入可因情境产生不同结果。 |
| 玩家体验 | **[工作假设]** 相较棋类，体验更依赖输入时序与纠错回路；可比较按键时序误差、落点方差和失败归因。 |

**[综合判断]** 这里的核心不是“移动速度更快”，而是目的地选择被控制信号和时间展开取代。若把整段跑跳都压成 `Position := destination`，会丢掉玩家实际控制的变量以及轨迹中碰撞的因果作用。

### 4.2 竞速：F-ZERO

**[来源事实]** 任天堂官方说明书分别列出油门、刹车、左右转向、重心偏移/侧滑与 Super Jet；界面显示速度、排名与能量，碰撞会消耗能量。说明书还提示打滑后松开油门恢复抓地，以及跳跃着陆会改变速度（[Nintendo，《F-ZERO》官方说明书，印刷页 5–9、13–14、29](https://www.nintendo.co.jp/clvs/manuals/common/pdf/CLV-P-SAABE.pdf)）。

| 层 | 案例分析 |
| --- | --- |
| 输入 | 油门、刹车、转向、侧滑、加速等并行控制。 |
| 行动 | 加速、制动、转向、改变重心等较小行动被组合为驾驶。 |
| 规则 | 车辆响应、赛道边界、碰撞、能量和圈数/排名目标共同结算。 |
| 状态变化 | 不只是位置；速度、朝向、能量、圈数和排名都可能变化。 |
| 空间关系 | 赛道是受边界约束的连续路线，方向与参照系不可省略。 |
| 系统动态 | 惯性、抓地、碰撞与连续输入产生车辆轨迹；排名是多车运动的关系结果。 |
| 玩家体验 | **[工作假设]** 掌控感更多来自对速度—朝向—赛道反馈的预测，而非选中终点；可用转向修正频率、碰撞率与速度曲线检验。 |

**[来源事实]** Sicart 在其原始研究中把电子游戏驾驶分析为由刹车、加速、转向、手刹等较小机制组成的复合机制，而不是一个不可分的“移动”按钮（[“Defining Game Mechanics”，Driving 例段](https://www.gamestudies.org/0802/articles/sicart)）。

**[综合判断]** 若分析尺度是“一场竞速中的玩家能力”，驾驶可以暂时作为机制；若尺度下钻到控制规则，它显然是机制组合。若只记录坐标差分，速度与朝向就会被误当成无关实现细节，实际却是后续状态的关键前因。

### 4.3 掷骰轨道桌游：《大富翁》

**[来源事实]** Hasbro 规则要求掷骰后沿箭头方向前进相应格数，并在落点执行购买、租金、税、抽牌等规则；经过或停在 GO 通常获得 200 美元。Chance/Community Chest 可命令棋子移动；被送进监狱时则“直接”移到 Jail，不经过 GO，也不收取 200 美元（[Hasbro，《Monopoly Vintage》官方规则，PDF 第 2 页](https://www.hasbro.com/common/instruct/Monopoly_Vintage.pdf)）。MDA 原始论文专门用骰子分布和 `Roll → Move` 反馈说明该游戏的机制与动态分析（[Hunicke 等，第 3 页](https://cdn.aaai.org/Workshops/2004/WS-04-04/WS04-04-001.pdf)）。

| 层 | 案例分析 |
| --- | --- |
| 输入 | 通常是“掷骰”决定是否启动流程；骰值由随机过程产生。 |
| 行动 | 玩家执行或系统结算“沿轨道前进 N 格”；目的地多数不是玩家选择。 |
| 规则 | 方向固定、距离由骰值决定；经过事件与落点事件不同；卡牌和监狱可覆盖通常流程。 |
| 状态变化 | 棋子位置、现金、资产/债务、监狱状态、回合流程等可能连续变化。 |
| 空间关系 | 有向环形轨道；路径中的 GO 与最终落点都具有语义。 |
| 系统动态 | 随机落点分布与经济交换长期累积，空间位移只是经济循环的路由器之一。 |
| 玩家体验 | **[工作假设]** 基础移动的能动性感受较弱，期待/风险感更多来自随机距离及落点后果；可比较掷骰前预测、路径选择变体与主观能动性评分。 |

**[边界/反例]** 同一个游戏明确区分“沿路径前进”与“直接移到监狱”。两者终态格相同也可能触发不同经过事件，因此移动不能一概约简为 `Position := destination`；至少还要说明路径是否可观察、是否触发经过规则、是行走还是直接重定位。

### 4.4 工人放置：Agricola（修订版）

**[来源事实]** Lookout 规则规定，在 14 个回合中玩家轮流把每个家庭成员放到一个未占据的行动格，并立即执行其行动；同一行动格不能容纳多个家庭成员。回合的 Returning Home 阶段再把家庭成员从行动格收回住房（[Lookout，《Agricola Revised Edition》官方规则，印刷页 4、6](https://www.lookout-spiele.de/upload/en_agricolare.html_Rules_Agricola-RE_EN.pdf)）。

| 层 | 案例分析 |
| --- | --- |
| 输入 | 玩家选择一个尚未放置的家庭成员与一个空行动格。 |
| 行动 | “放置工人并执行行动”，而不是沿棋盘路线旅行。 |
| 规则 | 行动格排他占据；放置后立即取资源或执行动作；阶段末统一回家。 |
| 状态变化 | 工人所在槽位、行动格占据、资源、农场状态和可用行动集合改变。 |
| 空间关系 | 行动格主要是功能槽位集合；通常没有从住房到行动格的经过路径或距离成本。 |
| 系统动态 | 轮流占位造成资源与机会竞争；“空间”组织的是行动可用性。 |
| 玩家体验 | **[工作假设]** 玩家更可能把它体验为预约/封锁行动，而非路程控制；可通过口述编码及加入距离规则后的策略变化检验。 |

**[综合判断]** 动画或自然语言可以说工人“移动到行动格”，但规则结构的关键是**选择、占据并触发槽位**。把它与棋子沿轨道前进归为同一个无差别机制，会遮蔽路径缺失、排他占位和立即执行行动这三个决定性特征。它既是“位置关系改变”的宽泛成员，也是“移动”标签可能过度扩张的对照案例。

## 5. 纵深案例：输入解析与耦合可达性

### 5.1 文字冒险/节点移动：Zork 与 Inform

**[来源事实]** Infocom 的原始 Zork 手册把方向词列为玩家在地图中移动的命令，并强调绘制地图的重要性（[Infocom，《Zork User Manual》原始手册扫描件](https://ifarchive.org/if-archive/infocom/shipped-documentation/Zork_PDP_11.pdf)）。Graham Nelson 后来创建的 Inform 作者工具则明确区分原始输入词与解析后的 action：输入 `s` 可被解析为 `[going south]`，action 是角色尝试执行的意图，可能成功或失败（[Inform，Actions](https://ganelson.github.io/inform-website/book/WI_7_1.html)）。`going` 行动又必须处理出发房间、目的房间、门、载具以及“没有出口”等情形（[Inform，Going from, going to](https://ganelson.github.io/inform-website/book/WI_7_13.html)；[Inform Standard Rules，Going](https://ganelson.github.io/inform/standard_rules/actns.html)）。这里并不主张 Zork 由 Inform 实现：Zork 材料提供历史作品的玩家侧命令，Inform 材料提供同类解析器后来可查验的规则形式化。

| 层 | 案例分析 |
| --- | --- |
| 输入 | `S`、`GO SOUTH` 等文本字符串。 |
| 行动 | 解析器规范化为 `going south` 的尝试；同义输入可产生同一行动。 |
| 规则 | 当前房间的方向关系映射到房间、门或无出口，再检查门、载具等条件。 |
| 状态变化 | 成功时当前 location/room 改变；失败时位置不变但可产生文本反馈或其他规则效果。 |
| 空间关系 | 有标签、有方向、可不对称的房间图；“相邻”是作者声明的边，不是几何距离。 |
| 系统动态 | 玩家通过反复尝试、记忆与制图形成对不可见图结构的认识。 |
| 玩家体验 | **[工作假设]** 导航负担取决于图的可预测性和反馈措辞，而不只取决于节点数；可测绘图错误率、重复命令和路径回忆。 |

**[综合判断]** 这个案例把四件常被压成一件的事拆开了：文字是输入，`going` 是行动，出口/门规则决定合法与结算，房间变化才是状态结果。图节点移动仍可由 `Move(actor, destination)` 表示其成功结果，但规则入口更接近 `TryGoing(actor, direction)`；目的地可能在解析和规则查询之后才被求出。

### 5.2 推箱子/空间解谜：《仓库番》

**[来源事实]** 《仓库番》官方站点把目标描述为把仓库中的货物运到指定存放点，并明确规定一次可推一个货物、不能推两个以上、不能拉货物（[Sokoban Official Site，“仓库番とは”](https://www.sokoban.jp/rule.html)）。Culberson 的原始技术报告进一步把问题形式化为二维网格中的推动者、箱子、障碍与目标：推动者不能与箱子或障碍共占格，只能推动单箱而不能拉，并证明一般化 Sokoban 是 PSPACE-complete；论文也把死锁视为求解的核心困难（[Culberson 1997，PDF 第 1–3 页](https://sokoban.dk/wp-content/uploads/2016/02/Sokoban-is-PSPACE-complete.pdf)）。

| 层 | 案例分析 |
| --- | --- |
| 输入 | 玩家给出一步方向输入。 |
| 行动 | 若前格为空则行走；若前格有箱且箱后格为空则尝试推动；否则失败/不动。 |
| 规则 | 占据、墙、一次一箱、只能推不能拉共同决定结算。 |
| 状态变化 | 普通步改变玩家位置；推动同时改变玩家与箱子两个位置；目标覆盖和未来可达性随之变化。 |
| 空间关系 | 离散正交网格；局部邻接与全局可达性同时重要。 |
| 系统动态 | 一个局部合法推动可能把箱子送进死角，永久删除未来解路径；回退能力取决于实现，而不是规则允许“拉”。 |
| 玩家体验 | **[工作假设]** 压力主要来自不可逆推动与未来可达性预测；可测死锁率、重开率、推动前停顿和撤销使用。 |

**[边界/反例]** 一次方向输入可能改变零个、一个或两个实体的位置；相同的局部位置差分还会改变全局可解性。若“一个机制只改变一个 actor 的 Position”，推动就被错误拆散；若把玩家步行和推箱都叫同一 `Move`，又会丢掉载荷、不可逆性与目标状态。

## 6. 边界案例：定义真正开始失真的地方

### 6.1 国际象棋的复合走子与历史权限

**[来源事实]** FIDE 把王车易位规定为王的一步，但该步把王移两格并把车转移到王刚经过的格；其合法性依赖王和相关车此前是否移动、路径占据、受攻击状态。吃过路兵只在对方兵刚刚两格前进后的紧接一手可用。三次重复中的“同一局面”也要求行棋方相同且可能走法相同；王车易位与吃过路兵权利会使棋子坐标看似相同的局面并不相同（[FIDE，第 3.8.2、3.7.3、9.2 条](https://handbook.fide.com/chapter/e012023)）。

**[边界/反例]** 这同时反驳两个过强命题：

- “一步移动只更新一个实体的位置”——王车易位一次更新王与车；
- “完整状态就是所有棋子坐标”——移动历史通过王车易位/吃过路兵权限继续影响合法行动集合。

**[综合判断]** 这里更适合把“王车易位”看作原子化的复合机制：对玩家和回合规则它是一手，对状态变换它含多个效果。原子性是分析尺度与事务边界，不等于内部不可分。

### 6.2 《大富翁》的直接入狱：相同终点、不同路径语义

**[来源事实]** 如前述官方规则所示，通常移动经过 GO 会收钱，而被送进 Jail 是直接移动且明确不经过 GO（[Hasbro 规则，PDF 第 2 页](https://www.hasbro.com/common/instruct/Monopoly_Vintage.pdf)）。

**[边界/反例]** 若两种结算都写成 `Position(token) := Jail`，系统将无法判断是否触发 GO。移动规则因此至少需要区分：遍历路径、瞬时重定位、经过触发和落点触发。终点相同并不保证机制等价。

### 6.3 bit Generations: ORBITAL：玩家不调用位移，系统却产生运动

**[来源事实]** Sicart 讨论的是任天堂 2006 年 Game Boy Advance 游戏 bit Generations: ORBITAL：玩家只调用吸引和排斥，借不同天体的引力场完成弹弓式飞行；作者正是用它追问所谓机制究竟是玩家动作、力场，还是最终的飞行（[Sicart，“Defining Game Mechanics”，第 31–33、69、84 段](https://www.gamestudies.org/0802/articles/sicart)）。任天堂后来推出的 Wii 改编版明确说明它在 2006 年原作上增加关卡和模式；其官方规则仍是用 A/B 键分别产生重力与反重力，控制玩家行星的轨道（[Nintendo，Art Style: ORBITAL 官方规则与操作](https://www.nintendo.co.jp/wii/wiiware/artstyle/wobj/rule/index.html)；[官方说明书，第 5 页](https://www.nintendo.co.jp/wii/wiiware/artstyle/pdf/man_wobj.pdf)）。后者只用于交叉核对这项共享控制关系，不把改编版的全部细节倒推给 GBA 原作。Sicart 同文把机制工作定义为代理为了与游戏状态交互而调用的方法。

| 层 | ORBITAL 暴露的分离 |
| --- | --- |
| 输入 | GBA 版本把两种机制映射到两个按键；Wii 改编版以 A/B 键触发。 |
| 行动 | 玩家调用吸引或排斥，而不是点选飞行终点。 |
| 规则 | 力场规则决定对象如何受影响。 |
| 状态变化 | 速度与位置随时间更新。 |
| 空间关系 | 连续空间中，玩家行星与周围天体的相对位置和距离决定受力与轨道。 |
| 系统动态 | 可见轨迹由连续规则迭代生成。 |
| 玩家体验 | **[工作假设]** 玩家可能强烈体验“操纵运动”，却没有直接调用名为移动的机制。 |

**[综合判断]** 这对“机制就是可见状态变化名称”的定义构成强边界。更精确的因果链是：玩家机制改变力/加速度条件，系统动态生成轨迹，轨迹再产生位置变化与体验。移动可以是机制的结果或动态，而不总是被玩家直接触发的机制。

### 6.4 Ludii：`move` 也可能完全不表示空间位移

**[来源事实]** Ludii 官方语言参考把 move ludemes 定义为在给定状态中生成合法 move 的构件，并区分玩家决定的 move 与由该决定产生的 effect moves；其 decision move 类型明确包括 `Pass`、`PlayCard`、`Propose` 与 `Vote`，其中 `pass` 的官方定义只是跳过当前回合（[Ludii Language Reference，第 8 章，尤其 §8.1.2–8.1.3、§8.2.16](https://ludii.games/downloads/LudiiLanguageReference.pdf)）。这是形式游戏描述语言中的“回合行动”用法，具体实例证明其中并不都包含空间位移。

**[边界/反例]** 英语 `move` 至少有“空间位移”和“一手合法行动”两种技术用法。若中文研究看到源码或论文中的 `move` 就自动编码成“移动机制”，会把出牌、放置、交换等非位移行动误收进移动家族。

## 7. 跨案例综合：移动在哪一层？

### 7.1 锚点公式的压力测试

用项目中的简化示例作为候选子型：

```text
Move(actor, destination)
requires Adjacent(Position(actor), destination)
effect Position(actor) := destination
```

| 案例 | `destination` 由玩家直接选？ | `Adjacent` 足够？ | 一次只改一个位置？ | 路径/时间不可忽略？ | 公式遗漏的关键 |
| --- | --- | --- | --- | --- | --- |
| 国际象棋普通走子 | 多数是 | 否 | 通常是 | 线路棋子要看路径 | 棋子可达关系、阻挡、王安全 |
| 《超级马力欧兄弟》 | 否 | 否 | 常为单角色但连续更新 | 是 | 控制信号、时序、碰撞 |
| F-ZERO | 否 | 否 | 多状态耦合 | 是 | 速度、朝向、能量、惯性 |
| 《大富翁》 | 通常否 | 轨道步可逐格解释 | 通常是 | 经过 GO 时是 | 随机距离、路径/落点触发、直接重定位 |
| Agricola | 是 | 无路径可言 | 是 | 通常否 | 排他槽位、行动触发、阶段归位 |
| Zork/Inform | 玩家选方向，不一定知道终点 | 图边意义上的相邻 | 通常是 | 路径是单条图边 | 解析、门、失败行动、隐藏图 |
| 《仓库番》 | 选方向而非终点 | 局部是 | 推动时否 | 长期可达性不可忽略 | 占据、耦合载荷、不可逆死锁 |
| ORBITAL | 否 | 否 | 多个运动变量 | 是 | 玩家控制力场，位移来自动态 |

**[综合判断]** 公式没有被“证伪”，因为项目已经要求在声明尺度上理解原语；它被更准确地限定为：

> **单实体、目的地已知、离散或可直接判定可达、一次结算的重定位机制。**

它适合教学起点，也适合许多回合制走子，但不应承担所有运动、导航、放置和驾驶现象的总定义。

### 7.2 尺度判定

| 声明的分析尺度 | “移动”的暂定归类 | 理由与使用条件 |
| --- | --- | --- |
| 状态/规则原子尺度 | 位置、方向、相邻、速度、占据、路径等是候选原语；`Relocate` 也可被某种 DSL 暂作原语 | 只有明确声明暂不分解，才能避免把自然语言词误当本体原子 |
| 单个游戏、具体规则尺度 | 走子、跳跃、驾驶、骰点前进、工人放置、going、推动是不同机制或复合机制 | 它们各有触发、合法性、结算和可观察状态变化 |
| 跨游戏目录尺度 | “移动”暂作机制家族 | 家族共享“改变实体相对空间/拓扑关系”，但内部要保留控制、路径、时间、载荷等差异轴 |
| 持续游玩组织尺度 | 导航、平台跳跃、竞速、空间封锁、推箱规划可成为玩法模板候选 | 需要多个机制围绕持续行动—决策—反馈编排；不能只因存在位置变化就判定 |
| 具体作品尺度 | 上述规则被关卡、数值、界面、叙事与资产实例化为游戏 | 同一机制家族在不同实现中可产生不同动态与体验 |

**[综合判断]** 因而，对题目的最短回答是：**“移动”依分析尺度而变；为了跨游戏研究的实用性，当前最稳妥的目录入口是机制家族，而不是无条件的单一原语或单一机制。** 这与项目四层主线并不冲突：家族是研究/索引组织方式，不是插入一层新的正式本体。

### 7.3 一个暂定的机制家族差异轴

以下差异轴用于记录变体，不宣称“最小”或“完整”：

| 差异轴 | 可观察取值示例 |
| --- | --- |
| 能动性来源 | 玩家选目的地；玩家选方向；玩家给连续控制；随机数决定；系统自动；其他实体强制 |
| 空间模型 | 方格/六角格；节点图；有向轨道；连续平面/三维空间；功能槽位；相对参照系 |
| 可达关系 | 邻接；射线且受阻挡；跳跃；门/边；动力学可达；规则直接重定位 |
| 时间模型 | 原子回合；逐格结算；分阶段；固定步长实时；连续/近连续模拟 |
| 路径语义 | 只看终点；经过格触发；轨迹碰撞；路径成本；路径不可观察；传送不经过 |
| 受控对象 | 单实体；编队；载具；玩家与载荷耦合；代理推动其他物；力场间接控制 |
| 合法性与成本 | 占据、阻挡、行动点、燃料/能量、历史权限、速度/转向约束、随机距离 |
| 结算效果 | 位置；位置+捕获；位置+触发行动；位置+速度/朝向；多实体位置；未来可达性 |
| 信息与反馈 | 完全可见目的地；隐藏图；预测轨迹；只给文本；误导或延迟反馈 |

**[开放问题]** 家族的最小共同条件是否应是“至少一个游戏实体相对于规则定义的空间/拓扑关系发生变化”？Agricola 的行动槽位和抽象状态机节点会把这个条件推得很宽；如果再把所有状态转移都叫“空间”，移动家族将失去区分力。后续需要用更多非空间状态机与纯放置案例反向校准。

## 8. 可快速原型化的设计变体与预测

这些变体不是改写正式定义，而是把上述争议转成可观察差异。每个原型应保留基线版本，以同关卡/同局面进行组内或交叉对照。

| 设计变体 | 最小规则改动 | 被隔离的变量 | **[工作假设]** 预测 | 建议观测 |
| --- | --- | --- | --- | --- |
| 1. 顺序王车易位 | 把王车易位拆成两手：先王两格，下一回合若仍合法再移车；其余规则不变 | 原子复合行动 vs 多步编排 | 中断机会会使易位显著变弱，玩家不再把它当一次“走子”；若体验与策略几乎不变，则原子事务边界的重要性被高估 | 易位选择率、中断率、局面评估差、玩家对“这是一手还是两手”的表述 |
| 2. 分叉轨道《大富翁》 | 保留骰值，在 2–3 个路口允许玩家选择等长支路；落点经济规则不变 | 随机距离 vs 路径能动性 | 路径选择会提高决策时间与主观能动性，并形成偏好路线；若选择迅速且分布近似随机，可能是分支后果不够可读 | 路口决策时长、路线熵、落点分布、主观能动性、优势路线集中度 |
| 3. “不直接入狱” | 被送入 Jail 时沿轨道逐格经过到 Jail，并照常结算经过 GO；另保留官方直接入狱基线 | 路径触发 vs 仅终点更新 | 即使终点相同，现金流与决策预期会明显改变，证明路径是规则语义而非动画细节 | GO 收入次数、破产时点、回合长度、玩家对传送/行走的区分 |
| 4. 邻接记忆式 Agricola | 保留全部原规则与回家阶段，只新增一条限制：同一家庭成员下一回合只能放到与其上回合行动格相邻的空格；第一回合不限，用小标记记录上回合格 | 无路径放置 vs 拓扑连续性 | 空间规划和连续路线会取代部分即时槽位估值；若策略几乎不变，则行动格图的拓扑约束过弱 | 路线重复率、因不相邻而放弃首选的次数、行动组合序列、资源效率 |
| 5. 《仓库番》一次拉箱券 | 每关允许一次把相邻箱子拉入玩家原格；其他规则不变 | 不可逆推动 vs 稀缺可逆性 | 死锁/重开减少，但难题会转成“何时消费拉箱券”；若只成为错误修复而无时机权衡，券太强或关卡未围绕它设计 | 死锁率、重开率、券使用时点、推动前停顿、解步数与推数 |
| 6. 文字冒险有成本的边 | 只新增一条规则：少数房间边消耗屏幕明确显示的时间/燃料；图、命令与出口反馈不变 | 无成本图邻接 vs 可见路径成本 | 可见成本会促成路线规划和绕路；若路线不变，成本可能太低或目标不要求资源权衡 | 重复边遍历、总成本、制图错误、路线回忆、成本更高边的选择率 |
| 7. F-ZERO“松油门即减速” | 只改变油门响应：松开油门后速度立即降到固定低速；转向、碰撞、赛道和目标不变 | 惯性速度保持 vs 直接速度响应 | 碰撞和修正输入会下降，但预测轨迹与保持速度的技能表达也下降；若高手差异不变，原赛道的主要挑战可能并非速度动态 | 速度曲线、油门切换次数、碰撞率、圈速方差、新手/熟手差距 |

原型共同记录表建议最少包含：初始状态、输入序列、解析行动、合法/失败结果、每次状态差分、路径/轨迹、决策时长、撤销/重开、完成时间。体验问卷只问可对应行为的问题，例如“结果在多大程度上由你的路线选择决定”，避免先向玩家灌输“这是移动机制”。

## 9. 对项目研究实践的暂定建议

以下只是本案例的**[综合判断]**，不修改项目正式文件：

1. 每次使用“移动”时顺手标明尺度与对象，例如“状态更新原语”“国际象棋走子机制”“驾驶复合机制”“跨游戏移动机制家族”；这遵循项目对声明尺度的要求，也符合驾驶可继续分解的来源证据（[`FOUNDATIONS.md`](../../theory/FOUNDATIONS.md)；[Sicart 的 Driving 例段](https://www.gamestudies.org/0802/articles/sicart)）。
2. 目录记录不要只设 `source/destination`；至少检查控制来源、拓扑、路径、时间、受控实体、结算与反馈。《大富翁》的经过/直接移动与 F-ZERO 的速度—朝向控制分别说明路径和时间不能总被省略（[Hasbro 规则](https://www.hasbro.com/common/instruct/Monopoly_Vintage.pdf)；[F-ZERO 官方说明书](https://www.nintendo.co.jp/clvs/manuals/common/pdf/CLV-P-SAABE.pdf)）。
3. 把 `input → parsed action → legality/resolution → state delta → runtime dynamics → experience hypothesis` 保留为证据链，尤其适用于实时游戏和文字解析器（[Inform：Actions](https://ganelson.github.io/inform-website/book/WI_7_1.html)；[MDA 原始论文](https://cdn.aaai.org/Workshops/2004/WS-04-04/WS04-04-001.pdf)）。
4. 把“移动过”和“现在在哪里”分开。王车易位权限、吃过路兵窗口、经过 GO、Sokoban 死锁都证明历史与路径可能进入完整状态（[FIDE 第 3、9 条](https://handbook.fide.com/chapter/e012023)；[Hasbro 规则](https://www.hasbro.com/common/instruct/Monopoly_Vintage.pdf)；[Culberson 1997](https://sokoban.dk/wp-content/uploads/2016/02/Sokoban-is-PSPACE-complete.pdf)）。
5. 允许同一具体规则在不同尺度换标签，但必须说明换尺度的理由；不要让“依尺度而变”成为回避定义的万能句（[`CONTEXT.md`](../../CONTEXT.md)）。
6. 先把“移动机制家族”当检索与比较工具。只有在更多案例中反复显示稳定边界后，再考虑是否需要正式化；当前开放研究阶段不应新增本体层级（[`ROADMAP.md`](../../ROADMAP.md)；[`ADR 0007`](../../docs/adr/0007-case-driven-research-spiral.md)）。

## 10. 开放问题

1. **[开放问题]** `Relocate(entity, destination)` 在项目的形式语言中应被写成原语、效果原语，还是最小机制？答案可能取决于形式语言要解释规则作者、运行时还是玩家。
2. **[开放问题]** “改变实体相对空间/拓扑关系”是否足以界定移动家族，还是还需要可达、路径或参照系条件，才能排除任意状态机转移？
3. **[开放问题]** 工人放置应是移动家族的外围成员、放置家族成员，还是用多标签表达两种相似性？单继承分类可能制造假冲突。
4. **[开放问题]** 当玩家操纵力场而系统生成轨迹时，目录应该索引玩家可调用机制、系统运动规则，还是两者及因果边？ORBITAL 说明只列一边会损失信息。
5. **[开放问题]** 路径是状态、事件序列还是仅为结算过程？《大富翁》的经过触发、实时碰撞和棋类线路检查可能需要不同表示。
6. **[开放问题]** 如何用统一但不过度简化的方式记录“未来可达性”这类二阶状态变化？《仓库番》的关键后果常不是当前坐标，而是解空间被永久缩小。
7. **[开放问题]** 玩家是否把工人放置、文字节点切换、直接传送和连续驾驶都称为“移动”，需要实证而非研究者直觉；应在后续原型中收集自然语言描述。
