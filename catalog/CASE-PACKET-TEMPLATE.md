### 表达视图

同一案例可以同时维护两个可追溯视图，但不制造两套互相冲突的事实：

- **教学最小视图**：刚好足以解释当前机制、设计问题和关键差异；优先使用自然语言、少量状态与一张核心机制卡。
- **研究充分视图**：足以复核合法性、边界、裁定或实现行为；按需展开完整状态、动作生命周期、调度、异常路径和证据账本。

教学最小视图必须指出省略了什么，并链接或定位到研究充分视图。研究包负责保存两者的追溯关系；未来书稿和网页可以选择性呈现。

## 3. 证据与陈述约定

本研究包使用以下标签：

- **[来源事实]**：来源直接支持的规则、实现、历史或设计陈述。
- **[项目定义]**：本项目为分析建立的操作性定义。
- **[观察]**：在明确场次、录像、日志或复现中实际出现的现象。
- **[综合判断]**：由多条证据推导的分析结论。
- **[工作假设]**：有待更多案例或观察检验的解释。
- **[反例]**：对当前主张构成边界或否证压力的材料。
- **[未知]**：目前证据不足；不是默认否定。

来源优先级取决于主张类型。官方规则或实现适合支持规则事实，却不能单独证明玩家通常怎样玩或有何体验；设计者陈述能说明意图，却不能单独证明意图已经实现；实际游玩材料只能支持其覆盖的版本、玩家和情境。

### 5.3 单项机制卡（每个核心机制复制一次）

#### M__: 机制名称

- **触发**（trigger）：什么事件或时间点使规则进入候选？
- **触发策略**：强制 / 可选 / 自动 / 条件队列 / 并发监听；多个触发怎样排序？
- **调度语义**：谁或什么使其获得运行机会；在哪个 tick、阶段、窗口或边界读取和提交？
- **行动者与能动性**：谁选择，谁只执行？
- **执行来源与裁定权**：谁实施、检查、解释和确认结果？两者不同时分别记录。
- **输入**：玩家或系统提供了什么信号或参数？
- **规则动作**：规则所辨认的动作是什么？不要直接把按键、手势与语义动作等同。
- **动作生命周期**：按需列出识别、声明、**锁定点**、作出、完成、确认、撤回或退出等里程碑；没有固定阶段时不要硬填。
- **前置条件**：该规则需要哪些状态事实？
- **合法性**：何时允许声明、进入队列、执行或完成？这四种“能”是否不同？
- **成本与决策锁定**：何时支付；哪些撤回、替换或修正权限在什么锁定点改变；资源、信息与后续资格怎样变化；失败是否返还？不要把这些后果压成一个“承诺原语”。
- **过程 / 结算**：确定计算、随机抽样、比较、搜索、物理模拟或裁定怎样进行？
- **行动结果**：区分声明结果、结算结果、状态效果、规则效果与玩家可感知反馈。
- **实体身份效果**：保留 / 创建 / 移除 / 替换 / 重定型 / 拆分 / 合并；不改变身份时可简写为“保留”。
- **效果时机与原子性**：立即 / 延迟 / 持续；能否打断；中间状态是否公开？
- **反馈**：系统怎样把变化传回参与者？反馈是否与真实状态一致？
- **规则修饰符**：哪些有类型的高阶规则改变触发、合法性、数值、目标选择、结算或效果？
- **状态附着物**：某实体当前携带、可移除或会到期的规则修改是什么？
- **来源定位**：
- **未知与争议**：

可选结构式：

```text
触发 + 前置条件 + 行动者/输入 + 规则动作
  → 过程/结算
  → 状态效果 + 规则效果 + 可感知反馈
```

结构式是可核对的速记，不是所有机制必须服从的最终语法。

## 7. 玩家层

规则描述可能的行动空间；玩家层描述人在具体情境中如何理解和利用它。两者证据不得混用。

### 7.1 **决策情境**与玩家活动

| 情境 | 可见状态与信念 | 可选行动 | 权衡 / 不可逆性 | 规则预期 / 已观察 / 推断 | 证据 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

只有当选择在玩家可获得的信息和目标下存在可说明的差异后果时，才把“有按钮可按”提升为**有意义决策**的证据。

### 7.2 推理、执行与协调

- 玩家需要感知、记忆、推断或预测什么？

- 多人之间怎样沟通、协调、威慑、承诺或欺骗？
- 哪些活动来自界面和媒介，而非抽象规则？

### 7.4 挑战、难度、平衡与体验

- **挑战**来自哪些任务要求？
- **难度**对哪些玩家、在什么配置和熟练度下成立？
- **平衡**具体指什么对象和指标？
- **公平**采用机会、程序、信息、结果还是补偿意义？
- 体验判断（紧张、满足、挫败、沉浸等）的情境和证据是什么？

以上均为带条件的主张，不把“规则看起来会导致”写成“玩家普遍感到”。

## 8. **玩法模板**候选

## 9. 从模板到这款**游戏**

这里分析实例化，不把完整游戏缩减为一张玩法模板。

- 角色绑定：抽象行动者、对象与关系由什么具体内容承担？
- 参数化：数值、概率、地图、牌组或生成条件怎样被设定？
- 内容与关卡：哪些具体内容塑造可达状态与学习顺序？
- 界面、材料与反馈：媒介怎样改变可见性、执行和理解？
- 模式与情境：哪些规则只在本配置成立？
- 多模板共存：这款游戏还实例化哪些玩法模板，怎样切换或叠加？
- 题材与表达：哪些意义来自内容和文化语境，而非机制结构？
- 单次游玩实例：哪些现象只属于某局、某群玩家或某次历史？

## 10. 跨案例比较

| 比较对象 | 判断 | 相同之处 | 关键差异 | 证据 |
| --- | --- | --- | --- | --- |
|  | 结构等价 / 行为等价 / 同一机制家族 / 功能类比 / 不可比 |  |  |  |

- **结构等价**：规则骨架在允许的重命名或参数变化后相同。
- **行为等价**：在声明的观察尺度上产生相同可观察行为，不要求内部结构相同。
- **机制家族**：共享关键结构特征，但允许有解释价值的系统差异。
- **功能类比**：在上层玩法中承担近似功能，不主张结构相同。

## 附录 A：原语候选准入检查

对每个候选分别回答；一次“看起来基础”不能代替检验。

1. 它能否跨多个明显不同的游戏重复出现？
2. 它是否承担稳定的表达职责，而不是某一题材的内容标签？
3. 若继续拆分，是否会失去当前分析所需的最小语义？
4. 它能否进入有类型的规则槽位，并有明确输入、输出或约束？
5. 它是否与参数值、具体内容、瞬时状态事实或高阶机制重名？
6. 引入它能否减少歧义或提高生成、比较、教学能力？
7. 是否存在反例，说明它只在某种媒介、传统或分析尺度成立？

结论只能是：暂准入 / 保留候选 / 降为类型、参数、内容或状态 / 拒绝。阶段一不宣称得到“完备原语表”。
