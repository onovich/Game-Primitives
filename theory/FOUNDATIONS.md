# 理论地基（草案）

本文档记录《游戏原语》的首版概念边界。它是一组待检验的操作性定义，而不是最终公理。

## 1. 研究对象

本项目研究的不是题材、画风或发行分类，而是游戏作为交互规则系统时的可描述结构：系统中有什么、它们处于何种状态、谁能在什么条件下做什么、这会造成什么变化，以及这些变化如何形成决策与体验。

## 2. 四个设计对象与三种组合关系

第一轮十二案校准后，项目保留下面这条面向读者的公开主线：

```text
原语 --语法--> 机制 --编排--> 玩法模板 --实例化--> 游戏
```

其中，**原语**、**机制**、**玩法模板**与**游戏**是四类主要设计对象；**语法**、**编排**与**实例化**是对象之间的组合关系，不是三个额外层级。这条主线是一把可以双向使用的设计缩放尺：向下拆解游戏，向上组合、设计和检验结构；它不是游戏必须遵循的创作顺序。

### 2.1 原语 Primitive

在某一描述尺度下，承担单一基础语义角色、暂不继续拆分的单位。

“原语”是**相对于模型和尺度**而言的。例如“资源”可以在高层分析中作为原语，也可以在更细模型中拆成数量、所有权、获取、储存和消耗规则。因此，是否为原语不能只凭词义判断，还要说明它在当前模型中的类型、接口和不可约理由。

候选原语家族包括：

- 实体与身份
- 状态与数值
- 关系与拓扑
- 空间与时间
- 动作、事件与效果
- 条件、约束与权限
- 信息、观察与隐藏
- 资源、价值与所有权
- 目标、评价与终止
- 随机性与不确定性

这些只是候选家族，不代表已经获得完备分类。

### 2.2 语法 Grammar：从原语到机制的关系

规定原语如何形成有效规则表达的组合约束。至少需要回答：

- 每个成分是什么类型，能占据什么角色？
- 动作由谁发起，作用于什么对象？
- 动作的前置条件和支付成本是什么？
- 动作发生后，哪些状态以何种顺序改变？
- 规则在什么时间、空间、信息和权限范围内有效？
- 多条规则如何顺序、并行、嵌套、互斥或反馈？

例如，“位置 + 相邻 + 移动”只有补全角色后才成为有意义的规则：

```text
Move(actor, destination)
requires Adjacent(Position(actor), destination)
effect   Position(actor) := destination
```

### 2.3 机制 Mechanic

由原语按语法构成、可被触发或反复运行的规则结构。机制不仅是一个动作名称，还应包含参与者、状态、条件、过程、效果、反馈和可作出的决策。

机制可以很小，例如抽牌；也可以由子机制组成，例如战斗结算。研究时必须声明分析尺度，避免把同一个词同时当作原语、机制和玩法模板。

### 2.4 编排 Orchestration：从机制到玩法模板的关系

规定一个或多个机制怎样通过顺序、并行、嵌套、门控、反馈、互斥、共享状态和目标等关系，组织成持续的玩家活动。**编排**不像规则语法那样追求单句式的强约束，但也不是把机制名称任意相加；它必须说明机制间关系、成立条件和反馈结构。

### 2.5 玩法模板 Gameplay Pattern / Archetype

多个机制围绕一种持续的玩家活动组织起来后形成的可复用结构。它通常包含：

- 核心决策或技巧
- 核心行动循环
- 资源与风险结构
- 反馈和成长节奏
- 失败、恢复与终局方式
- 局内与局外的关系

“塔防”“卡组构筑”“银河城”可以作为待分析的玩法模板；“动作”“角色扮演”等更宽泛的市场类型未必天然对应单一结构。

### 2.6 实例化 Instantiation：从玩法模板到游戏的关系

规定玩法模板怎样在具体规则版本、内容、界面、媒介、技术、叙事、美学和社会情境中成为可实现、可游玩的作品结构。**实例化**不表示一款游戏只能来自一个模板，也不表示作品、规则版本和一次实际游玩是同一个研究对象。

### 2.7 游戏 Game

一个被具体实现和体验的交互系统。游戏不等于玩法模板；它还包含内容、美学、界面、叙事、技术条件、商业结构、玩家社群及游玩情境。相同模板可以成为不同游戏，同一游戏也可以容纳多个模板。

## 3. 基本规则句

首版规则句采用下面的骨架：

```text
在 Context 中，
当 Trigger 发生且 Preconditions 成立时，
Actor 可以或必须对 Target 执行 Action，
支付 Cost，
依 Resolution 得到 Effects，
系统产生 Feedback，
并据此更新 Goals / Evaluation。
```

并非每条规则都需要显式填写全部槽位；省略项必须能由上下文确定。未来需要研究哪些槽位不可缺少，以及不同机制家族是否需要不同句式。

## 4. 组合不只是相加

机制组合至少可能具有以下关系：

- **顺序**：机制 A 的结果成为机制 B 的输入。
- **并行**：多个机制同时争夺注意力或资源。
- **嵌套**：一个机制发生在另一个机制的阶段或作用域内。
- **门控**：机制 A 决定机制 B 是否可用。
- **反馈**：机制结果改变下一轮同类机制的参数。
- **互斥**：选择一个机制会关闭另一个机制。
- **转换**：同一状态在两个机制中承担不同语义角色。
- **耦合**：机制共享状态，因此彼此产生非局部影响。

玩法设计的关键往往不在“用了哪些机制”，而在这些关系如何塑造选择、节奏和后果。

## 5. 当前开放问题

1. 原语应以语义不可约、实现不可约，还是设计操作不可约为标准？
2. 是否存在跨电子游戏、桌面游戏、体育、角色扮演与儿童游戏的共同原语集合？
3. 机制的同一性由规则结构、玩家体验还是设计意图决定？
4. 玩法模板与类型、核心循环、设计模式之间如何划界？
5. 涌现玩法能否由静态语法充分表达？
6. 界面、身体技能、社交约定和元游戏应进入哪一层？
7. 一种描述语言如何兼顾可读性、可计算性与设计价值？

这些问题将通过文献研究、跨游戏案例、反例、原型和形式化实验逐步回答。
