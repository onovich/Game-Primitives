# 游戏原语

**Game Primitives**

《游戏原语》是一个面向游戏设计基础结构的长期研究与写作项目。它试图回答一个朴素但困难的问题：**游戏由什么构成，这些构成部分又如何组合成可玩的系统？**

项目沿着下面的路径展开：

```text
原语 ──语法──> 机制 ──编排──> 玩法模板 ──实例化──> 游戏
```

- **原语（Primitive）**：描述游戏所需的基础语义单位，例如实体、状态、关系、动作、事件、约束、资源、信息、时间与目标。
- **语法（Grammar）**：规定原语可以怎样连接，包括类型、角色、前置条件、作用域、顺序、因果和状态变化。
- **机制（Mechanic）**：由原语按一定语法构成、能够反复运行的规则结构。
- **玩法模板（Gameplay Pattern / Archetype）**：多个机制围绕核心决策、行动和反馈形成的可复用组织方式。
- **游戏（Game）**：玩法模板在具体内容、界面、美学、技术、叙事与社会情境中的实现。

这条链条不是单向生产线。真实游戏会反过来检验机制分类、语法假设和原语边界。

## 核心目标

1. 建立一套可讨论、可修订的游戏原语词汇与类型系统。
2. 梳理原语的组合语法，使“位置 + 相邻 + 移动”不再只是关键词堆叠，而能写成有条件、有作用对象、有结果的规则句子。
3. 解释经典机制如何成立、如何变化、如何失效，以及它们之间如何组合或冲突。
4. 研究机制如何形成玩法模板，玩法模板又如何在具体作品中成为完整游戏。
5. 将研究沉淀为一本既保持严谨、又能被设计师和普通玩家读懂的书。

## 核心立场

> 游戏不是由类型标签直接构成的。类型是表象；其下是玩法模板、机制、组合语法与原语。

“所有游戏都能由有限原语表达”目前是本项目的**研究假设**，不是预先宣布的结论。原语是否真正不可再分、同一机制是否存在唯一分解、不同媒介是否共享同一语法，都需要通过案例、反例和形式化工作持续检验。

我们的工作原则是：

- 先求可解释，再求可枚举。
- 先写清组合关系，再罗列术语。
- 每个抽象都应回到具体游戏，每个案例也应能反过来修正抽象。
- 同时保留自然语言解释、结构化表示和形式化表达的可能性。
- 明确区分已有研究、项目定义、工作假设与个人判断。

## 预期成果

- 一部《游戏原语》书稿。
- 一套游戏原语、语法、机制与玩法模板的术语体系。
- 一组跨类型的经典游戏拆解与对照案例。
- 一种便于人类阅读、也可能被工具处理的描述记法。
- 面向机制设计、原型验证、程序生成和 AI 辅助设计的方法。
- 一个用于浏览、讨论和公开研究过程的早期研究网站，以及成稿阶段重新设计的正式网页版。

## 早期约定：先研究，后成稿

项目当前不以尽快写出章节为目标。接下来可以进行大量讨论、阅读、案例拆解、概念试验与形式化探索，允许材料暂时重复、矛盾或没有结论。书稿目录首先是一张**问题地图**，不是要求依次填满的写作清单。

早期可以通过 GitHub Pages 发布一个“研究版”网站，用于阅读、检索、串联和分享仓库中的阶段性成果。这个网站应当清楚标示内容状态，并尽量从仓库材料生成，避免维护两份正文。它服务于研究，不代表最终书籍的结构、文风或视觉形态。

进入成稿阶段后，我们会围绕读者的理解路径重新组织和改写材料，并重新设计正式网页版；早期网站不构成其版式和信息架构约束。完整阶段约定见 [`ROADMAP.md`](ROADMAP.md)。
